# `/docs` endpoint capability notes

This file is the source text for the FastAPI Swagger UI description served at
`GET /docs`. It describes only the capabilities implemented by the current
project code and validation records. Runtime model capability discovery still
comes from `GET /v1/models`; clients must not treat this page as a replacement
for model discovery.

## What `/docs` is

- `GET /docs` serves FastAPI's autogenerated Swagger UI.
- `GET /openapi.json` serves the OpenAPI document consumed by Swagger UI.
- `GET /redoc` serves FastAPI's autogenerated ReDoc page.
- `GET /docs/oauth2-redirect` is FastAPI's Swagger OAuth redirect helper.
- `/docs` is documentation only. It does not load models, unload models,
  transcribe audio, expose GPU state, or prove that a backend has passed
  runtime validation.

## Implemented HTTP API surface

The current FastAPI app registers these business endpoints:

| Method | Path | Implemented behavior |
| --- | --- | --- |
| `GET` | `/health` | Returns service status, version, host name, and best-effort local CUDA GPU visibility. |
| `GET` | `/v1/models` | Lists declared models and their current runtime status and capabilities. By default this is the two Qwen3-ASR models; MOSS is listed only when explicitly enabled. |
| `GET` | `/v1/models/{model_id}/status` | Returns one model's lifecycle state, active request count, backend, token limit, timestamps, and VRAM field. |
| `POST` | `/v1/models/{model_id}/load` | Loads a declared model through the selected supported backend. |
| `DELETE` | `/v1/models/{model_id}` | Unloads one model immediately or schedules unload after active requests finish. |
| `DELETE` | `/v1/models` | Applies unload behavior to all declared models. |
| `POST` | `/v1/audio/transcriptions` | Accepts one uploaded audio file, validates requested capabilities, normalizes/splits audio, transcribes, merges results, and returns JSON, verbose JSON shape, plain text, or `202` queued job for audio over the sync threshold. |
| `POST` | `/v1/audio/transcription-jobs` | Creates an in-memory asynchronous transcription job in a single-worker FIFO queue. |
| `GET` | `/v1/jobs/{job_id}` | Returns queued/running/completed/failed/cancelled job state, queue position, progress, result, or error. |
| `DELETE` | `/v1/jobs/{job_id}` | Cancels queued jobs immediately or requests cancellation at the next running job phase/chunk boundary. |
| `POST` | `/v1/audio/alignments` | Exists only as a reserved endpoint and currently returns `422 capability_not_supported`. |

## Declared model capabilities

`GET /v1/models` declares these model IDs by default:

- `qwen3-asr-1.7b`
- `qwen3-asr-0.6b`

For both models, the declared capability set is:

- `transcription: true`
- `backends: ["transformers"]`
- `streaming: false`
- `timestamps: []`
- `forced_alignment: false`
- `diarization: false`
- `segment_timestamps: false`
- language values as returned by `/v1/models`
- Chinese dialect values as returned by `/v1/models`

The default model is controlled by `ASR_DEFAULT_MODEL` and defaults to
`qwen3-asr-1.7b`. MOSS cannot be selected as the default model in this
release.

When `ASR_ENABLE_MOSS=1` is set, `/v1/models` additionally declares:

- `moss-transcribe-diarize-0.9b`

Its first-release capability set is deliberately conservative:

- `transcription: true`
- `backends: ["transformers"]`
- `streaming: false`
- `timestamps: []`
- `forced_alignment: false`
- `diarization: true`
- `segment_timestamps: true`
- language values: `auto`, `zh`, `en`

For MOSS, `response_format=verbose_json` returns parsed
`segments[].start`, `segments[].end`, `segments[].speaker`, and
`segments[].text` when the model output is parseable. The `timestamps`
request parameter still does not support `word` or `char` timestamps, and MOSS
segment timestamps are not treated as forced alignment.

## Transcription request behavior

Implemented form fields for `POST /v1/audio/transcriptions` and
`POST /v1/audio/transcription-jobs` are:

- `file`: required single audio upload.
- `model`: optional; defaults to the configured default model.
- `language`: defaults to `auto`.
- `response_format`: `json`, `text`, or `verbose_json`.
- `timestamps`: accepts `none`, `word`, or `char` syntactically, but only
  `none` is supported by declared capabilities today.
- `backend`: accepts `auto`, `transformers`, or `vllm` syntactically, but only
  `auto` resolving to `transformers` and explicit `transformers` are supported
  by declared capabilities today.
- `max_new_tokens`: optional; default is `512`, hard limit is `4096`.
  MOSS uses a model-specific default of `2048` when the field is omitted.
- `context`: optional adapter prompt context, combined with hotwords and limited
  to `4000` characters.
- `hotwords`: optional JSON string array or comma-separated string, merged into
  adapter context.
- `split_strategy`: `auto`, `none`, `fixed`, `silero`, `energy`, or `vad`.
- `max_chunk_seconds`: optional per-request chunk upper bound.
- `overlap_seconds`: optional chunk overlap.
- `preserve_segments`: optional flag to return chunk-level details.

Audio uploads are bounded by `ASR_MAX_UPLOAD_MB`, default `512`.
Each uploaded audio file is bounded to `21600` seconds, or 6 hours.
Synchronous requests over `600` seconds of detected audio duration are converted
to asynchronous jobs and return `202` with `status_url`.

## Async job behavior

The job manager is in-memory and process-local:

- Queueing is FIFO.
- Only one job runs at a time.
- `ASR_MAX_QUEUED_JOBS` defaults to `20`.
- Job result TTL defaults to `3600` seconds.
- A process restart loses in-memory jobs.
- Progress reports real service phases and chunk completion counts; it does not
  invent token-level or frame-level Qwen progress inside a chunk.
- Uploaded job files and temporary work directories are cleaned after completed,
  failed, or cancelled jobs.

## Runtime modes

The service can run with two adapter modes:

- `ASR_ADAPTER=mock` is the default local development mode and does not require
  CUDA, torch, Transformers, or a model cache.
- `ASR_ADAPTER=qwen` uses the lazy Qwen adapter and requires the WSL CUDA torch
  environment plus HF native Transformers dependencies to be installed and
  validated separately.

MOSS is gated independently:

- `ASR_ENABLE_MOSS=0` or unset keeps MOSS out of `/v1/models`.
- `ASR_ENABLE_MOSS=1` adds `moss-transcribe-diarize-0.9b`.
- With `ASR_ADAPTER=mock`, all declared models use mock inference for local API
  tests.
- With `ASR_ADAPTER=qwen ASR_ENABLE_MOSS=1`, Qwen model IDs dispatch to the
  Qwen adapter and the MOSS model ID dispatches to the MOSS adapter.

The public model list is the same shape in both modes. Successful real Qwen
transcription requires the WSL GPU environment, not just the existence of
Swagger UI. Successful real MOSS transcription additionally requires the WSL
MOSS smoke test and dependency validation.

## Explicitly not implemented or not declared

These items must not be described as available capabilities:

- WebSocket streaming transcription. No
  `/v1/audio/transcriptions/stream` route is registered.
- vLLM as a declared first-release backend. The registry does not declare
  `vllm`; requests for it return
  `422 capability_not_supported`.
- SGLang Omni as a first-release serving path. It is not integrated into this
  gateway and must not be exposed on a LAN port by this service.
- Timestamps. `timestamps=word` and `timestamps=char` return
  `422 capability_not_supported`.
- Forced alignment. `/v1/audio/alignments` always returns
  `422 capability_not_supported`.
- Cross-chunk speaker identity stitching for MOSS. When audio is split into
  chunks, MOSS speaker labels are chunk-local and the response warnings include
  `moss_speaker_labels_are_chunk_local`.
- Multiple-file transcription in one request. The implemented routes accept a
  single `file` upload.
- Database, Redis, Celery, distributed workers, multi-worker inference, public
  tunnel exposure, MiMo-V2.5-ASR, or Web UI.

## Adversarial review checklist

Use this checklist when changing `/docs`, `/openapi.json`, or this file:

- If an option appears in a Pydantic/FastAPI schema only because the parser
  accepts its string value, verify whether runtime capability validation also
  permits it.
- If a capability is not returned by `GET /v1/models`, do not document it as
  supported.
- If a future branch exists in adapter code but the registry does not declare
  it, document it as unavailable.
- If a feature requires WSL CUDA validation, distinguish code support from
  environment readiness.
- Keep port references limited to the service API port `18080`; do not add
  historical test ports or expose worker ports.
